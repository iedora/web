# Shared infra — `infra/`

The single deploy entry point. One Tofu root provisions the Hetzner VPS, every Cloudflare resource, the GitHub Actions config, and every Docker container on the box. Root `AGENTS.md` covers cross-cutting conventions.

## What this owns

**Tofu state (`infra/tofu/`):**

- **Hetzner VPS** (`hetzner.tf`) — `hcloud_server.iedora` (CX23, Falkenstein, x86_64) + SSH key + firewall.
- **Cloudflare resources** (`main.tf`) — 3 R2 buckets (`backups`, `observability`, menu assets minted in the menu-local root), their scoped tokens, DNS records for `menu.iedora.com` + `auth.iedora.com` + `obs.iedora.com` (all grey-cloud A records pointing directly at the VPS IPv4; Caddy terminates TLS on-box).
- **GitHub Actions config** (`github.tf`) — `github_actions_secret.secrets[*]` + `github_actions_variable.vars[*]`, `for_each` over a locals map; values flow from BWS via `TF_VAR_*` aliases.
- **Zitadel app state is NOT in Tofu.** The Zitadel container itself is here (`containers.tf::module.zitadel`), but the application-level configuration (org, project, OIDC app, roles, action targets, PAT) is reconciled by `bin/zitadel-apply` (Stage 3 of the pipeline) — see `infra/cmd/zitadel-apply/`. Tofu no longer carries a `zitadel.tf` or the `zitadel/zitadel` provider.

**Tofu-managed SHARED containers (`containers.tf`):** every always-on Docker container on the VPS is declared here via `kreuzwerker/docker` over SSH:

- `infra-postgres` — Postgres 18, shared by menu + zitadel databases.
- `infra-backups` — daily `pg_dumpall` → R2, GPG-encrypted.
- `infra-openobserve` — OTLP receiver + UI on `:5080`, R2 cold tier.
- `infra-zitadel` + `infra-zitadel-login` — Zitadel IdP + its login UI (share the `zitadel-bootstrap` named volume).
- `infra-caddy` — TLS termination + reverse proxy, bound to the VPS public IPv4.

The menu app (`infra-menu-web`) is NOT here. It's owned by Stage 4
(`task deploy:menu` / `iedora deploy menu`) via the `dockerOnHetzner`
productRuntime — SSH + docker pull/run, env composed from BWS + Tofu
outputs. Caddy routes to `infra-menu-web` by network alias, so the
container can come and go between deploys without touching Caddy.

Containers attach to `docker_network.iedora` (`name = "iedora"`); container-DNS is by alias (`infra-postgres`, `infra-zitadel`, …) and the network name itself is internal.

**Self-built images (`infra/backup/`):** Dockerfile + `backup.sh` / `restore.sh` / `run.sh`. Rebuilt only when bumping Postgres major.

## Hard rules

1. **Declarative-first.** Every resource here is Tofu-managed. **Edit `.tf` files, never the upstream UI** — Tofu's unconditional reconciliation will silently clobber UI edits on the next `task up`.
2. **Tofu-managed credentials write through to BWS.** For resources like `cloudflare_api_token.workers_deploy`, the value is generated by Tofu and pushed to BWS by `task up`. Editing BWS directly is wasted work — the next apply restores Tofu's value. See `docs/deploy.md` § Secret rotation.
3. **Bootstrap order is BWS → Tofu → write-through.** Populate the bootstrap secrets in BWS, then `task up` provisions everything else AND writes any Tofu-minted credentials back to BWS. See `docs/deploy.md`.
4. **Follow `docs/terraform-style.md` when editing any `.tf`.** Pessimistic `~>` pins, `for_each` over `count`, `validation` blocks, predictable naming.
5. **State file is encrypted in git.** PBKDF2 + AES-GCM, passphrase from `IAC_BOOTSTRAP_STATE_PASSPHRASE`. Rotation via the `fallback` block migration — see `docs/deploy.md` § Secret rotation.
6. **Run the pre-merge runbook on every deploy-shape change.** Touching anything under `infra/cmd/iedora/`, `infra/cmd/with-secrets/`, `infra/cmd/bws-upsert/`, `infra/cmd/zitadel-apply/`, `infra/cmd/menu-db-migrations/`, `infra/cmd/openobserve-dashboards/`, `infra/internal/{tlsprobe,r2,cloudflare,bws}/`, `infra/tofu/*.tf`, `infra/bin/*`, `Taskfile.yml`, or `products/*/infra/tofu/*.tf` requires running the 6-step `task down`/`task up` sequence in [`docs/deploy.md`](../docs/deploy.md) § Pre-merge runbook before merge. Takes ~30–40 min and spins real cloud — but it's the only thing that catches the class of bugs (DNS races, R2 empty-then-delete, state-vs-cloud drift, one-shot reveal recovery) that unit tests can't.

## File layout

```
infra/
  justfile                              thin shims for deploy / destroy / doctor (→ bin/iedora) +
                                        day-2 ops (logs / console / backup / restore / wipe-postgres /
                                        build-backup / rotate-secret / zitadel-rebootstrap)
  cmd/iedora/                           Go orchestrator behind iac / app / deploy / destroy /
                                        pipeline / doctor. Stage 3 configurator registry +
                                        Stage 4 productRuntime registry live here. The
                                        cert-issuer probe (LE vs Caddy internal) lives in
                                        internal/tlsprobe. See docs/deploy.md § Failure modes
                                        for the post-mortem catalogue.
  bin/iedora                            Go-run entrypoint for the orchestrator. NOT
                                        auto-layered through bin/with-secrets — the Taskfile
                                        composes the right --stage per task.
  bin/with-secrets                      BWS wrapper. Operator only needs `BWS_ACCESS_TOKEN` in shell; the wrapper discovers `BWS_PROJECT_ID` via `bws project list` and `CLOUDFLARE_ACCOUNT_ID` via CF /accounts API, then exports every BWS secret as `TF_VAR_*`. No on-disk `infra/.env`.
  tofu/                                 single encrypted root
    versions.tf                         providers: hcloud, cloudflare ~> 5.19, github ~> 6.12,
                                        kreuzwerker/docker ~> 3.7 + encryption block
                                        (zitadel provider removed — Stage 3 handles app state)
    variables.tf                        bootstrap creds + GH config + hostnames + container secrets
    hetzner.tf                          hcloud_server.iedora + firewall + SSH key
    main.tf                             R2 buckets + DNS (menu/auth/obs)
    containers.tf                       docker_network.iedora + docker_volume.zitadel_bootstrap +
                                        every SHARED docker_container (postgres, backups,
                                        openobserve, zitadel, zitadel-login, caddy). The menu
                                        container lives in Stage 4, NOT here.
    secrets.tf                          random_password.* (postgres, backup_passphrase, zitadel
                                        masterkey, …) + IAC_* BWS write-through
    github.tf                           Tofu-managed GH Actions secrets + variables
    outputs.tf                          hetzner_ipv4 + menu_* env outputs for Stage 4
  postgres/
    init.sql                            CREATE DATABASE menu / zitadel (runs once on first init)
  backup/                               self-built Postgres-backup image
    Dockerfile, run.sh, backup.sh, restore.sh
```

## Commands

The Taskfile at the repo root is the operator entry point. Each pipeline
stage has its own task; `bin/iedora` is the underlying Go binary that
each task shells through `bin/with-secrets --stage <s>`.

```
task up               # full pipeline: infra:up → app:apply → deploy:all
task down             # full teardown: products → infra:down

task infra:up         # Stage 2 only: tofu apply on infra/tofu/
task infra:down       # Stage 2 destroy
task app:apply        # Stage 3: every configurator (Zitadel via bin/zitadel-apply)
task deploy:menu      # Stage 4: docker pull + run on the box (menu)
task deploy:house     # Stage 4: bun build + tofu apply on per-product root (house)
task deploy:all       # Stage 4: fan out every product in parallel

task dev              # local dev stack (infra/cmd/dev)
task dev:down         # tear it down
task dev:reset-db -- menu      # scoped per-DB reset
task doctor           # preflight: PATH, BWS auth, bootstrap secrets

task bws -- <cmd>     # exec any command with BWS secrets hydrated (stage=iac by default)
task zitadel:grants   # re-run iedora-admin grants (bin/zitadel-apply --grants-only)
```

Day-2 ops (logs, psql, backup, restore, secret rotation, postgres wipe,
zitadel rebootstrap) are operator-side raw SSH:

```
HOST=$(infra/bin/with-secrets --stage iac -- tofu -chdir=infra/tofu output -raw hetzner_ipv4)
ssh root@$HOST docker logs -f --tail=200 infra-<svc>             # logs
ssh -t root@$HOST docker exec -it infra-postgres psql -U postgres
ssh root@$HOST docker exec infra-backups sh /backup.sh           # pg_dump now
ssh -t root@$HOST docker exec -it infra-backups sh /restore.sh   # restore
```

`bin/iedora` is a thin Go-run wrapper. Subcommands: `iac apply|destroy`,
`app apply`, `deploy [products…]`, `destroy [products…]`, `pipeline [-d]`,
`doctor`. The Taskfile owns the `bin/with-secrets --stage X` composition
per task — invoke `bin/iedora` directly only for stage-agnostic
subcommands (`doctor`, `--help`).
